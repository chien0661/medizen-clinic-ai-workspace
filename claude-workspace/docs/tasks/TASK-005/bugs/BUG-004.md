# BUG-004: Cross-Clinic Undo Merge Succeeds — No Clinic Ownership Check on Undo

**Type:** Business Logic Bug / Data Integrity Issue
**Severity:** Critical
**Status:** RESOLVED
**Feature:** TASK-005 - Patient Management
**Found By:** Test Agent
**Date:** 2026-04-27

## SRS Reference
- Multi-tenant isolation: operations on clinic resources must be restricted to users of the same clinic.
- Undo merge is a privileged operation on a `patient_merge_log` record scoped to a clinic.

## Detail Design Reference
- `app/modules/patients/services/merge_service.py` — `undo_merge()` function.
- `undo_merge()` fetches the `PatientMergeLog` by primary key (`db.get(PatientMergeLog, merge_id)`) without validating that `merge_log.clinic_id` matches the calling user's JWT clinic.

## Test Scenario
- **File:** `tests/integration/patients/test_patients_negative.py`
- **Scenario:** `TestUndoNegativePaths::test_undo_merge_from_different_clinic_returns_404_or_403`
- Clinic-B admin creates a merge_log; Clinic-A admin successfully calls `POST /api/v1/patients/merge/{clinic_b_merge_log_id}/undo`.

## Expected Behavior
- A user belonging to Clinic-A must NOT be able to undo a `patient_merge_log` that belongs to Clinic-B.
- Expected response: **403 Forbidden** or **404 Not Found**.

## Actual Behavior
- The undo succeeds with **200 OK**: `{"merge_log_id": "...", "message": "Merge undone successfully. Both patient records are now active."}`
- A Clinic-A user can restore Clinic-B's soft-deleted patient records and modify Clinic-B's merge history.

## Steps to Reproduce
1. Create Clinic-A user + Clinic-B user (both with `patient.merge` permission).
2. Clinic-B user: create keep_patient and drop_patient in Clinic-B.
3. Clinic-B user: `POST /api/v1/patients/merge` → get `merge_b_log_id`.
4. Clinic-A user: `POST /api/v1/patients/merge/{merge_b_log_id}/undo`.
5. Observe `200 OK` — **cross-tenant undo succeeded**.

## Database State
```sql
-- After the cross-clinic undo, Clinic-B's drop_patient is restored:
SELECT id, is_deleted FROM patient WHERE id = '<clinic_b_drop_patient_id>';
-- Result: is_deleted = false  (restored by Clinic-A user — WRONG)
```

## Impact
- **Critical Security Impact:** Any authenticated user with `patient.merge` permission can modify another clinic's patient records and merge history, breaking multi-tenant isolation.
- **Data Integrity:** Clinic-B's patient soft-deletion state is modified without authorization.
- **Compliance Impact:** Violates multi-tenant data isolation requirements.
- **Audit Trail:** The undo audit log will show the wrong `user_id` / `clinic_id` performing the undo.

## Suggested Fix
In `app/modules/patients/services/merge_service.py`, `undo_merge()`, add a clinic ownership check after fetching the merge_log:

```python
merge_log = await db.get(PatientMergeLog, merge_id)
if merge_log is None or merge_log.is_deleted:
    raise NotFoundError(f"Merge log {merge_id} not found")

# ADD THIS: verify calling user's clinic matches the merge_log's clinic
# The calling clinic_id must be obtained from the request context
# and passed into undo_merge() as a parameter.
if merge_log.clinic_id != calling_clinic_id:
    raise NotFoundError(f"Merge log {merge_id} not found")  # Use 404 to avoid enumeration
```

Also update the API route to pass `clinic_id` from the JWT into `undo_merge()`.

Alternatively: scope the `db.get()` to a filtered SELECT that includes `clinic_id`:
```python
result = await db.execute(
    select(PatientMergeLog).where(
        PatientMergeLog.id == merge_id,
        PatientMergeLog.clinic_id == calling_clinic_id,
    )
)
merge_log = result.scalar_one_or_none()
if merge_log is None:
    raise NotFoundError(f"Merge log {merge_id} not found")
```

### Root Cause

`merge_service.undo_merge()` fetched `PatientMergeLog` by primary key only (`db.get(PatientMergeLog, merge_id)`), with no check that `merge_log.clinic_id` matched the calling user's clinic. Any authenticated user with `patient.merge` permission could undo merges belonging to any clinic.

### Changes Made

1. Added `clinic_id: UUID | None = None` parameter to `undo_merge()` in `app/modules/patients/services/merge_service.py`.
2. After the `db.get()` fetch, added a post-fetch ownership check:
   ```python
   if clinic_id is not None and merge_log.clinic_id != clinic_id:
       raise NotFoundError(f"Merge log {merge_id} not found")  # 404 prevents ID enumeration
   ```
3. Updated the `undo_merge` route in `app/modules/patients/api/routes.py` to extract `clinic_id = _require_clinic_id()` and pass it to the service call.

`clinic_id` is optional (default `None`) to maintain backward compatibility with existing unit tests that mock `db.get()` and call the service directly without a clinic context. The production API route always supplies `clinic_id`.

### Verification

`test_undo_merge_from_different_clinic_returns_404_or_403` passes (404 response). All 6 pre-existing `TestUndoMerge` unit tests still pass. 117/117 non-perf tests pass.
