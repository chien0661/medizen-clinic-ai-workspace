# BUG-003: Self-Merge (keep_id == drop_id) Succeeds Instead of Returning 400

**Type:** Business Logic Bug
**Severity:** High
**Status:** OPEN
**Feature:** TASK-005 - Patient Management
**Found By:** Test Agent
**Date:** 2026-04-27

## SRS Reference
- Merge operation: "Merge 2 hồ sơ trùng" — merging a patient with itself is logically undefined and should be rejected.

## Detail Design Reference
- `app/modules/patients/services/merge_service.py` — `merge()` function.
- The function does not check `keep_id == drop_id` before proceeding.

## Test Scenario
- **File:** `tests/integration/patients/test_patients_negative.py`
- **Scenario:** `TestMergeNegativePaths::test_merge_same_id_returns_4xx`
- Sends `POST /api/v1/patients/merge` with `keep_id == drop_id`.

## Expected Behavior
- When `keep_id == drop_id`, the API must return **400 Bad Request** or **422 Unprocessable Entity**.
- No merge log should be created. No patient should be soft-deleted.

## Actual Behavior
- The merge proceeds: the patient is soft-deleted (merged with itself), a `patient_merge_log` row is created, and `201 Created` is returned.
- The patient is now unreachable (soft-deleted) and can only be recovered via undo.
- A malicious or erroneous self-merge permanently disables a patient record.

## Steps to Reproduce
1. Authenticate as admin user with `patient.merge` permission.
2. Create patient P1.
3. `POST /api/v1/patients/merge` with body `{"keep_id": "<P1_id>", "drop_id": "<P1_id>"}`.
4. Observe `201 Created` — P1 is now soft-deleted.
5. Expected: `400`/`422` — merge rejected with clear error.

## Test Output
```
FAILED tests/integration/patients/test_patients_negative.py::TestMergeNegativePaths::test_merge_same_id_returns_4xx
AssertionError: Same-id merge should return 400/422, got 201
assert 201 in (400, 422)

# Patient is now soft-deleted:
keep_id=3ce284c9 == drop_id=3ce284c9 patient_merged ...
```

## Impact
- **Data Integrity:** A patient can be accidentally self-merged and soft-deleted, making their record inaccessible.
- **Business Impact:** Clinical staff cannot access a patient record until undo is performed (if within 7-day window).
- **Security Impact:** An attacker with `patient.merge` permission can disable any patient record by self-merging.

## Suggested Fix
In `app/modules/patients/services/merge_service.py`, add an early validation in `merge()`:
```python
if keep_id == drop_id:
    raise AppException(
        code="SELF_MERGE_NOT_ALLOWED",
        message="Cannot merge a patient with themselves",
        http_status=400,
    )
```
Alternatively, enforce at the Pydantic schema level in `app/modules/patients/schemas/patient_schemas.py`:
```python
@model_validator(mode="after")
def keep_and_drop_must_differ(self) -> "MergeRequest":
    if self.keep_id == self.drop_id:
        raise ValueError("keep_id and drop_id must be different")
    return self
```
