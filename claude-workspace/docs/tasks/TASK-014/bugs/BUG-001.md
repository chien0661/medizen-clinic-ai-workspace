# BUG-001: RLS Tenant Isolation Not Enforced at HTTP Layer (BYPASSRLS on app DB user)

**Task**: TASK-014  
**Severity**: CRITICAL — Security Defect  
**Found by**: Test Agent (tenant isolation contract tests)  
**Date**: 2026-04-27  
**Status**: OPEN  

---

## Summary

Row-Level Security (RLS) policies are defined and enabled on all 5 HR tables (`shift_template`, `shift`, `recurring_schedule`, `leave_request`, `time_log`), but they are **never enforced** because the application connects to PostgreSQL as the `cms` user, which is a **superuser with `BYPASSRLS=True`**.

As a result, User B from Clinic B can:
- **PATCH** `shift_template` records belonging to Clinic A → gets HTTP 200 (should be 403/404)
- **PATCH** `shift` records belonging to Clinic A → gets HTTP 200 (should be 403/404)
- **Approve** leave requests from Clinic A → gets HTTP 200 (should be 404)
- **Check in** using a `shift_id` from Clinic A → succeeds (should be 404/403)

---

```
current_user=cms  session_user=cms

```

## Root Cause

The `shift_service.get_shift_template()`, `get_shift()`, and `leave_service.get_leave_request()` all use `db.get(Model, id)` (SQLAlchemy `Session.get()` by primary key). This lookup is **not filtered by `clinic_id`** at the service layer. The expectation was that RLS would reject the lookup if the record belongs to a different clinic — but since `cms` has BYPASSRLS, RLS never fires and the full record is returned regardless of clinic context.

**Affected service functions:**
- `shift_service.get_shift_template(db, template_id)` → `db.get(ShiftTemplate, template_id)` — no clinic filter
- `shift_service.get_shift(db, shift_id)` → `db.get(Shift, shift_id)` — no clinic filter
- `leave_service.get_leave_request(db, leave_id)` → `db.get(LeaveRequest, leave_id)` — no clinic filter
- `recurring_service.get_recurring_schedule(db, schedule_id)` → `db.get(RecurringSchedule, schedule_id)` — no clinic filter
- `attendance_service.check_in()` → `db.get(Shift, shift_id)` — only checks `user_id == user_id` but NOT `shift.clinic_id == clinic_id` for the clinic

---

## Impact

**Security boundary broken**: Any authenticated user in one clinic can read and mutate HR records from any other clinic by guessing or enumerating UUIDs.

- Data leakage: Sensitive employee scheduling/leave data accessible cross-clinic
- Data corruption: Cross-clinic shift/leave mutations corrupt another clinic's HR state
- GDPR/compliance: Personal data (attendance logs, leave reasons) accessible to unauthorized tenants

---

## Remediation Options

Two complementary fixes should BOTH be applied:

### Fix 1 (Preferred — Defense in Depth): Change DB user to non-superuser

Create a restricted DB role `cms_app` with no superuser / no BYPASSRLS for the application connection. Configure via `DATABASE_URL` environment variable.

```sql
CREATE ROLE cms_app LOGIN PASSWORD '...' NOSUPERUSER NOCREATEDB NOCREATEROLE INHERIT;
GRANT CONNECT ON DATABASE cms TO cms_app;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO cms_app;
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO cms_app;
-- RLS will now be enforced for cms_app
```

### Fix 2 (Service layer — belt-and-suspenders): Add clinic_id check to all get_* functions

In each `get_*` helper, add an explicit `clinic_id` assertion after fetching by PK:

```python
async def get_shift_template(db: AsyncSession, template_id: UUID, clinic_id: UUID | None = None) -> ShiftTemplate:
    tmpl = await db.get(ShiftTemplate, template_id)
    if tmpl is None or tmpl.is_deleted:
        raise NotFoundError("ShiftTemplate not found")
    if clinic_id is not None and tmpl.clinic_id != clinic_id:
        raise NotFoundError("ShiftTemplate not found")  # Don't reveal existence
    return tmpl
```

Then pass `clinic_id` from the route handler via `_clinic_id()`.

**For `check_in`**: the existing code checks `shift.user_id != user_id OR shift.clinic_id != clinic_id` which is correct — but the test shows it's returning HTTP 200. The condition is:
```python
if shift.user_id != user_id or shift.clinic_id != clinic_id:
    raise ForbiddenError(...)
```
This correctly raises `ForbiddenError` when `shift.clinic_id != current_clinic_id`. However, the test uses user B with clinic B's JWT. When user B calls check-in, `_clinic_id()` should return clinic B's ID but the shift belongs to clinic A. This should raise ForbiddenError → 403. That this test fails (gets 200) suggests `_clinic_id()` may not be properly set in the test context, OR the existing clinic_id check in check_in IS working but the test's assertion of the scenario is wrong. Needs further investigation.

---

## Immediate Workaround

Until the DB role is changed, **Fix 2** (service-layer clinic_id check) provides partial protection by preventing cross-clinic mutations even when RLS is bypassed.
