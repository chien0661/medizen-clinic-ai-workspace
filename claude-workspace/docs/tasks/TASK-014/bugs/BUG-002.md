# BUG-002: PATCH /shifts/{id} Allows Inverted Times (end_time ≤ start_time)

**Task**: TASK-014  
**Severity**: MAJOR  
**Found by**: Test Agent (BR-01 business rule tests)  
**Date**: 2026-04-27  
**Status**: OPEN  

---

## Summary

`PATCH /api/v1/shifts/{id}` with `{"start_time": "13:00:00"}` on a shift that currently has `end_time: "12:00:00"` returns HTTP 200 and silently stores the shift with `start_time > end_time`. The same issue affects `PATCH /api/v1/shift-templates/{id}`.

The unit test `TestShiftUpdateRejectsInvertedTimes::test_shift_update_rejects_inverted_times` passes (mocked), but the service logic is **not triggered correctly** at the API level when only a partial field is patched.

---

## Evidence

```
PATCH /api/v1/shifts/{id}  {"start_time": "13:00:00"}
Response 200: {"start_time":"13:00:00","end_time":"12:00:00",...}
```

Expected: HTTP 400 with BUSINESS_RULE_VIOLATION.

Test: `test_br01_patch_shift_to_inverted_times_rejected` → **FAIL** (got 200)  
Test: `test_br01_patch_shift_template_equal_times_rejected` → **FAIL** (got 200)

---

## Root Cause Analysis

In `shift_service.update_shift()`:

```python
for key, val in fields.items():
    if val is not None and hasattr(shift, key):
        setattr(shift, key, val)
if shift.end_time <= shift.start_time:
    raise BusinessRuleError("end_time must be after start_time")
```

The check `val is not None` skips the field when `val` is `None` — but `time(0, 0)` is a valid falsy-ish value. More importantly:

The PATCH payload `{"start_time": "13:00:00"}` is processed via `ShiftUpdate.model_dump(exclude_none=True)` which produces `{"start_time": time(13, 0)}`. The service sets `shift.start_time = time(13, 0)`.

Then `if shift.end_time <= shift.start_time` checks `time(12, 0) <= time(13, 0)` → **True** → should raise `BusinessRuleError`.

However, the response shows the mutation succeeds. This suggests the exception is being swallowed or the `db.flush()` rollback doesn't prevent the commit. The API route handler may not be wrapping the `BusinessRuleError` → HTTP 400 translation correctly for this specific path.

Further investigation needed: trace the exception handler mapping for `BusinessRuleError` in `app/core/exceptions.py`.

## Remediation

1. Verify the `BusinessRuleError` exception handler maps to HTTP 400 in `app/core/exceptions.py`.
2. Add explicit flush + exception catching in the PATCH route handler.
3. Alternatively, validate the combined (new_start, existing_end) or (existing_start, new_end) in `ShiftUpdate` schema using `model_validator` — but this requires both fields to be present, which may not always be the case in partial patches.
4. Best fix: in `update_shift()` and `update_shift_template()`, raise the `BusinessRuleError` BEFORE the flush, so it happens in a clean state.
