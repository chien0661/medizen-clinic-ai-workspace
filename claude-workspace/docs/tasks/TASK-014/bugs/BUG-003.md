# BUG-003: Self-Approval of Leave Request Not Rejected at API Level

**Task**: TASK-014  
**Severity**: MAJOR  
**Found by**: Test Agent (BR-07 business rule tests)  
**Date**: 2026-04-27  
**Status**: OPEN  

---

## Summary

`POST /api/v1/leave-requests/{id}/approve` by the same user who submitted the leave request returns HTTP 200 instead of 400/403. The unit test `TestApproveLeaveRejectsSelfApproval` passes (using mocked service layer), but at integration level the check is bypassed.

---

## Evidence

```
User A submits POST /leave-requests  → lr_id
User A then calls POST /leave-requests/{lr_id}/approve (same user)
Response: 200 {"status":"approved","approved_by":"<user_A_id>"}
```

Expected: HTTP 400 (BusinessRuleError: "Cannot approve your own leave request")

Test: `test_br07_self_approval_rejected_400` → **FAIL** (got 200)

---

## Root Cause Analysis

`leave_service.approve_leave_request()` at line 90 in `leave_service.py`:

```python
if lr.user_id == approved_by:
    raise BusinessRuleError("Cannot approve your own leave request")
```

This check exists. However, the user who submits the leave is `_user_id()` from JWT, and the approver is also `_user_id()` from JWT. These SHOULD be equal for self-approval.

The unit test passes because it mocks `leave_service.get_leave_request` to return a mock where `mock_lr.user_id = user_id` and calls with `approved_by=user_id`.

The integration failure suggests one of:
1. `lr.user_id` is not set correctly on creation (maybe the FK resolution returns a different type)
2. The `approved_by` from `_user_id()` is a different object/type than `lr.user_id` from the DB
3. The UUID comparison fails silently (e.g., `UUID != str(UUID)`)

Looking at the response: `"approved_by":"a5865c28-06bf-4901-9bf3-675ac2d78f4c"` matches the `user_id` in the leave request — so the approval succeeded. The check IS evaluated but must be returning False.

Likely cause: **UUID type comparison failure** — `lr.user_id` from the DB may be a `uuid.UUID` object, but `approved_by` from `_user_id()` could be a `str` if the middleware returns a string UUID. Check `current_user_id.get()` return type in `app/core/db.py`.

---

## Failing Tests

- `tests/integration/test_hr_business_rules.py::TestBR07SelfApprovalLeave::test_br07_self_approval_rejected_400`

---

## Remediation

1. In `approve_leave_request()`, ensure comparison uses consistent types:
   ```python
   if str(lr.user_id) == str(approved_by):
       raise BusinessRuleError("Cannot approve your own leave request")
   ```
2. Or ensure `_user_id()` always returns `uuid.UUID` (not `str`).
3. Add an integration test that directly invokes the service (not mocked) to catch this type mismatch earlier.

---

## Resolution

**Status**: RESOLVED in commit `cc5c539`
**Verified**: All originally-failing tests now pass (Round 2 verification, 2026-04-28)
**Fix summary**: UUID type mismatch fixed by coercing both sides to `str` before comparison (`str(lr.user_id) == str(approved_by)`) in `leave_service.py`; self-approval check now placed before the status check to ensure it fires in all pending-state scenarios.
