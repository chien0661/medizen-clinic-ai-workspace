# BUG-004: check_in Allows Soft-Deleted Shift and Cross-User Shift at API Level

**Task**: TASK-014  
**Severity**: CRITICAL — Security Defect  
**Found by**: Test Agent (BR-09, BR-10 business rule tests)  
**Date**: 2026-04-27  
**Status**: OPEN  

---

## Summary

`POST /api/v1/attendance/check-in` with a `shift_id` that is either:
- **Soft-deleted** (the shift has `is_deleted=True`) — returns 201 instead of 404
- **Owned by another user in the same clinic** — returns 201 instead of 403

Both are security/data integrity violations.

---

## Evidence

### BR-09: Soft-deleted shift accepted

```
Admin creates shift → Admin soft-deletes shift (DELETE /shifts/{id} → 204)
Staff calls POST /attendance/check-in {"shift_id": <deleted_shift_id>}
Response: 201 (check-in created, linked to deleted shift)
```

Expected: HTTP 404 "Shift not found"

### BR-10: Cross-user shift check-in

```
Admin creates shift assigned to admin user (user_id = admin_id)
Staff calls POST /attendance/check-in {"shift_id": <admin_shift_id>}
Response: 201 (check-in created for staff, linked to admin's shift)
```

Expected: HTTP 403 "does not belong to you"

---

## Root Cause Analysis

In `attendance_service.check_in()`:

```python
shift = await db.get(Shift, shift_id)
if shift is None or shift.is_deleted:
    raise NotFoundError("Shift not found")
if shift.user_id != user_id or shift.clinic_id != clinic_id:
    raise ForbiddenError("Cannot check in to a shift that does not belong to you")
```

**For BR-09 (deleted shift)**: `db.get(Shift, shift_id)` uses SQLAlchemy identity map / L1 cache. If the shift was deleted in the SAME test session/transaction context before check-in, the ORM may return the cached (non-is_deleted) version. The `is_deleted` flag may not be freshly loaded from DB after the DELETE API call.

**For BR-10 (cross-user)**: `shift.user_id != user_id` should catch this. But if `shift.user_id` is a `uuid.UUID` and `user_id` is a `str` UUID (or vice versa), the comparison returns `False != True` → the condition `False OR False` → ForbiddenError is NOT raised.

This is the same UUID type mismatch issue as BUG-003. The `user_id` parameter in `check_in()` comes from `_user_id()` in the route, which reads from `current_user_id.get()`.

## Remediation

### BR-09 (Deleted shift):

Use a fresh DB query instead of `db.get()` to avoid identity map caching:

```python
result = await db.execute(
    select(Shift).where(Shift.id == shift_id, Shift.is_deleted.is_(False))
)
shift = result.scalars().first()
if shift is None:
    raise NotFoundError("Shift not found")
```

### BR-10 (Cross-user):

Normalize UUID types before comparison:

```python
if UUID(str(shift.user_id)) != UUID(str(user_id)) or UUID(str(shift.clinic_id)) != UUID(str(clinic_id)):
    raise ForbiddenError("Cannot check in to a shift that does not belong to you")
```

Or ensure `_user_id()` always returns a `uuid.UUID` object, not a string.
