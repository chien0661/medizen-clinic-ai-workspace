# BUG-001: Account Lockout DB Update Rolled Back (is_locked Never Persists)

**Task:** TASK-003
**Found By:** Test Agent
**Date:** 2026-04-27
**Type:** Integration
**Severity:** Critical
**SRS Reference:** BA §22 — Account lockout after N failed attempts
**Detail Design Reference:** `app/modules/auth/services/lockout_service.py:_lock_user`
**Test File:** `tests/integration/test_auth_lockout_real_db.py::TestLockoutFlowRealDB::test_lockout_end_to_end`

---

## Expected Behavior

After 5 consecutive failed login attempts for the same user:

1. `User.is_locked` = `True` is committed to DB.
2. A 6th login attempt (any password) returns HTTP `423 Locked`.
3. An `audit_log` row with `action='user.locked'` is committed.

## Actual Behavior

1. `User.is_locked` remains `False` in DB — the update is always rolled back.
2. A 6th login attempt returns HTTP `401` (proceeds as if user is not locked).
3. The `audit_log` row for `user.locked` is also rolled back.
4. **Redis counter (`lockout:{clinic_id}:{username}`) correctly increments** — the counter logic works.

## Root Cause

In `app/modules/auth/services/auth_service.py`, the `login()` function:

1. Calls `record_failed_attempt()` (in `lockout_service.py`) → Redis INCR → **if count >= threshold**, calls `_lock_user()`.
2. `_lock_user()` sets `user.is_locked = True`, calls `db.flush()`, and writes an audit row via `write_audit()` — **all within the existing SQLAlchemy session/transaction**.
3. After `record_failed_attempt()` returns, `login()` raises `ValueError("invalid_credentials")`.
4. FastAPI's `get_db` dependency catches the exception and issues `session.rollback()`.
5. **Both** the `failed_login_count` update AND the `is_locked=True` update (plus the audit row) are undone by the rollback.

The Redis counter is incremented outside the DB transaction (in a separate Redis connection) so it persists correctly. But the DB state is lost on every failed attempt.


## Fix Options (for Code Implementation Agent)

### Option A: Autonomous Transaction (Recommended)
Use a new independent DB connection/session for the lock update, committed before `login()` raises:

```python
async def _lock_user(
    engine,   # pass the SQLAlchemy async engine
    user_id: UUID,
    clinic_id: UUID,
    username: str,
) -> None:
    """Lock user in an autonomous transaction (separate from the failing login tx)."""
    from sqlalchemy.ext.asyncio import async_sessionmaker
    factory = async_sessionmaker(engine, expire_on_commit=False)
    async with factory() as autonomous_session:
        user = await autonomous_session.get(User, user_id)
        if user is None or user.is_locked:
            return
        user.is_locked = True
        autonomous_session.add(user)
        await autonomous_session.flush([user])
        await write_audit(
            autonomous_session,
            action="user.locked",
            entity_type="User",
            entity_id=user_id,
            new_data={"reason": "max_failed_login_attempts", "username": username},
        )
        await autonomous_session.commit()
```

### Option B: Explicit Commit Before Raise
In `login()`, commit the session before raising `ValueError` (breaks current transaction model — not recommended since audit rows for the failed attempt would also commit).

### Option C: Deferred Lockout Check
Don't lock in the DB at all during failed attempts; instead, check the Redis counter at login time and return 423 if Redis count >= threshold. Removes DB dependency for lockout but means `is_locked` field is not used for initial lockout (only for admin-locked accounts).

**Recommended: Option A** — cleanest, least disruptive, keeps the DB as the source of truth for `is_locked`.

## Impact

**Critical — core security AC not met:**
- AC: "Login sai 5 lần → account locked, return 423 Locked" is NOT satisfied.
- A malicious actor can attempt unlimited login retries against any account (Redis counter limits to window, but DB lock never triggers → 423 never returned).
- Only Alembic schema has `is_locked` column; the column is never set to `True` via the normal lockout flow.

## Status

FIXED — commit `b2aa08c` on branch `feature/task-003-auth`.

**Fix applied (2026-04-26):** `_lock_user()` refactored to use an autonomous transaction via `AsyncSessionLocal()`. The function no longer accepts a `db` session parameter; it opens its own independent session, commits `is_locked=True` + audit row, then closes. The caller's session can still roll back without affecting the lock.
