### Issue 1 (CRITICAL): Test fixture hardcodes Docker-only hostname — CI test job will fail
**File:** `tests/conftest.py:17`
**Severity:** CRITICAL

**Current Code:**
```python
TEST_DATABASE_URL = "postgresql+asyncpg://cms:cms@postgres:5432/cms"
```

**Problem:**
The hostname `postgres` only resolves inside the docker-compose network. In GitHub Actions CI (`.github/workflows/ci.yml:59`), `DATABASE_URL` is set to `postgresql+asyncpg://cms:cms@localhost:5432/cms`, but `conftest.py` ignores that env var and uses the hardcoded value. The pytest job will fail to connect to Postgres on the CI runner.

The implementation agent claims "14/14 tests pass" — that holds only when run inside the api container, not in CI.

**Fix:**
```python
import os
TEST_DATABASE_URL = os.environ.get(
    "DATABASE_URL",
    "postgresql+asyncpg://cms:cms@localhost:5432/cms",
)
```

**Rationale:** Single source of truth for DB URL — env var first, sensible local default. Same code works in Docker (env injected by compose), in CI (env injected by workflow), and on the host (uses default).

### Issue 3 (MAJOR): Unhandled-exception handler logs nothing — every 500 is a black box
**File:** `app/core/exceptions.py:117-123`
**Severity:** MAJOR

**Current Code:**
```python
async def unhandled_exception_handler(request: Request, exc: Exception) -> ORJSONResponse:
    return _error_response(
        request,
        code="INTERNAL_SERVER_ERROR",
        message="An unexpected error occurred.",
        http_status=500,
    )
```

**Problem:**
Any non-`AppException` (DB integrity error, AttributeError, network timeout, …) is converted to a generic 500 with **zero logging**. In production we'd have no traceback, no exception type, no context — only the request_id in the response body, with nothing to correlate it to in the logs.

**Fix:**
```python
from app.core.logging import get_logger
log = get_logger(__name__)

async def unhandled_exception_handler(request: Request, exc: Exception) -> ORJSONResponse:
    log.exception(
        "unhandled_exception",
        path=request.url.path,
        method=request.method,
        exc_type=type(exc).__name__,
    )
    return _error_response(
        request,
        code="INTERNAL_SERVER_ERROR",
        message="An unexpected error occurred.",
        http_status=500,
    )
```

**Rationale:** structlog is already wired in lifespan; `.exception()` captures the traceback as `exc_info`. Without this every 500 in production is unactionable.

### Issue 4 (MAJOR): No `MetaData(naming_convention=...)` — autogenerated constraints will violate §2.1
**File:** `app/core/base_model.py:12-15`
**Severity:** MAJOR

**Problem:**
Design §2.1 mandates `ix_<table>_<cols>`, `uq_<table>_<cols>`, `fk_<table>_<col>_<reftable>`, `pk_<table>`. The 0001 migration hand-names its index/PK correctly, but:

1. The `TenantMixin.clinic_id` foreign key has no explicit name — when reflected into Alembic autogenerate, SQLAlchemy will produce something like `clinic_clinic_id_fkey` (Postgres default), which fails the convention.
2. Every future task using `--autogenerate` will produce off-spec names and force manual rewrites.

This is the single highest-leverage fix on the foundation: set it once, every downstream task gets it for free.

**Fix:**
```python
from sqlalchemy import MetaData

NAMING_CONVENTION = {
    "ix": "ix_%(table_name)s_%(column_0_N_name)s",
    "uq": "uq_%(table_name)s_%(column_0_N_name)s",
    "fk": "fk_%(table_name)s_%(column_0_name)s_%(referred_table_name)s",
    "pk": "pk_%(table_name)s",
    "ck": "ck_%(table_name)s_%(constraint_name)s",
}

class Base(DeclarativeBase):
    metadata = MetaData(naming_convention=NAMING_CONVENTION)
```

Also add `render_as_batch=False` (already implicit) and consider `include_schemas=True` to env.py if multi-schema is on the roadmap (it isn't — skip for now).

**Rationale:** §2.1 compliance is checkable by autogenerate; setting the convention at the metadata level means every model in TASK-002..TASK-024 conforms automatically.

### Issue 5 (MAJOR): structlog ProcessorFormatter chain is missing `format_exc_info`
**File:** `app/core/logging.py:35-42`
**Severity:** MAJOR

**Current Code:**
```python
formatter = structlog.stdlib.ProcessorFormatter(
    foreign_pre_chain=shared_processors,
    processors=[
        structlog.stdlib.ProcessorFormatter.remove_processors_meta,
        structlog.processors.ExceptionRenderer(),
        renderer,
    ],
)
```

**Problem:**
`ExceptionRenderer` reads the already-rendered `exception` key but does not promote `exc_info` from the stdlib `LogRecord` into a structured exception dict. Without `structlog.processors.format_exc_info` (or `dict_tracebacks`) before it, calls like `log.exception(...)` from foreign loggers will not produce a full traceback in JSON output. Combined with Issue 3, our 500 logging would be effectively useless.

**Fix:**
```python
processors=[
    structlog.stdlib.ProcessorFormatter.remove_processors_meta,
    structlog.processors.format_exc_info,   # promote exc_info → exception dict
    structlog.processors.dict_tracebacks,    # JSON-friendly tracebacks
    renderer,
],
```

`shared_processors` should also include `structlog.processors.format_exc_info` so that calls via the structlog API produce the same shape.

**Rationale:** Without proper exception serialization, JSON log shippers (Loki, Datadog, …) can't parse the traceback as a single field.

### Issue 6 (MAJOR): Session-scoped `event_loop` fixture is deprecated and slated for removal
**File:** `tests/conftest.py:20-25`
**Severity:** MAJOR

**Current Code:**
```python
@pytest.fixture(scope="session")
def event_loop():
    loop = asyncio.new_event_loop()
    yield loop
    loop.close()
```

**Problem:**
`pyproject.toml` pins `pytest-asyncio>=0.24`. As of 0.23, redefining `event_loop` is deprecated and emits `DeprecationWarning`; in a future minor release it will be removed. The recommended replacement is `asyncio_default_fixture_loop_scope = "session"` in `[tool.pytest.ini_options]`.

The implementation note says "tested: passes without warning" — the warning is suppressed only because `asyncio_mode = "auto"` happens to interact with it; this will break on the next pytest-asyncio bump.

**Fix:**
1. Delete the `event_loop` fixture.
2. In `pyproject.toml`:
```toml
[tool.pytest.ini_options]
asyncio_mode = "auto"
asyncio_default_fixture_loop_scope = "session"
testpaths = ["tests"]
```
3. Remove the redundant `@pytest.mark.asyncio` decorators from `tests/integration/test_health.py` (auto mode handles them).

**Rationale:** Eliminate deprecation tech debt now while the surface area is small.

### Issue 9 (MINOR): `request_id` is regenerated on every error response (no middleware sets it)
**File:** `app/core/exceptions.py:94`
**Severity:** MINOR

**Problem:**
`request_id = str(getattr(request.state, "request_id", uuid4()))` falls back to a fresh UUID because no middleware sets `request.state.request_id`. The same logical request, if it triggers two error responses (it won't here but will in upload/streaming flows later), would have two different IDs. More importantly, the request_id never appears in any log line — operators can't correlate.

**Fix (defer if you prefer, but document):**
Add a tiny middleware in TASK-002 (or now):
```python
@app.middleware("http")
async def request_id_middleware(request: Request, call_next):
    rid = request.headers.get("X-Request-ID", str(uuid4()))
    request.state.request_id = rid
    structlog.contextvars.bind_contextvars(request_id=rid)
    try:
        response = await call_next(request)
        response.headers["X-Request-ID"] = rid
        return response
    finally:
        structlog.contextvars.unbind_contextvars("request_id")
```

**Rationale:** Without this the response shape’s `request_id` is non-actionable. Acceptable to defer to TASK-002 explicitly — but please add a NOTE in code/handoff.

### Issue 11 (NIT): `pkgutil.walk_packages` in alembic env.py — works for now, watch for side-effects later
**File:** `alembic/env.py:19-24`
**Severity:** NIT

**Observation:**
The auto-import via `pkgutil.walk_packages` filtering on `".models"` substring is pragmatic for the foundation but has two latent risks worth knowing:

1. Any module path that happens to contain `.models` literally as a substring (e.g. a future `app.modules.imports.models_helper`) will be imported even if it's not a model module.
2. If a model module has import-time side effects (registers signal handlers, opens connections), `alembic upgrade head` will trigger them.

**Suggestion:** Tighten the predicate to `pkg_info.name.endswith(".models") or "/models/" in pkg_info.name.replace(".", "/")` or, when modules grow, switch to an explicit registry.

**Rationale:** Acceptable for TASK-001 — flag for revisit when ≥ 5 modules exist.
